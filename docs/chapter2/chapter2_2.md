## 项目名称

跨平台智能座舱电子手册与车辆助手

## 项目目标

本项目用于支撑 Android 跨平台开发实验课程。它不是单一技术栈的演示 Demo，而是一套统一业务规格。课程中的 Android 原生、Web/PWA、Hybrid、React Native、Flutter、Kotlin Multiplatform、Compose Multiplatform 等实现，都应尽量围绕同一份需求交付，从而保证最终比较具有可解释性。

项目模拟智能座舱中的电子手册和车辆助手场景。用户可以在横屏车机界面中浏览车辆功能说明，搜索关键内容，收藏常用条目，查看浏览历史，在离线状态下访问缓存内容，并查看模拟车辆数据。

## 目标用户

- Android 与跨平台开发学习者。
- 需要理解多技术栈工程差异的移动端开发者。
- 需要做跨平台技术选型的技术负责人、架构师或课程学习小组。

## 使用场景

### 场景一：浏览车辆电子手册

用户进入应用后，左侧显示目录树，右侧显示图文说明。用户可以点击目录切换不同章节，例如空调、车灯、驾驶辅助、座椅、充电、保养等。

### 场景二：搜索车辆功能

用户输入关键词，例如“胎压”“空调”“蓝牙”“充电”，应用展示匹配的手册条目。点击搜索结果后进入对应详情，并高亮或定位到相关内容。

### 场景三：收藏和历史

用户可以收藏常用说明，也可以查看最近浏览过的条目。收藏和历史应在应用重启后保留。

### 场景四：离线使用

用户在无网络或主动开启离线模式后，仍可访问已经缓存的手册内容、图片和收藏记录。未缓存内容应有清晰的空状态或错误提示。

### 场景五：车辆助手

应用展示模拟车辆数据，例如车速、续航、胎压、电量、车门状态、空调状态等。数据可以由本地模拟器、Android Service 或共享模块提供。

### 场景六：调用原生能力

用户可以通过页面入口触发文件选择、相机、通知等原生能力。不同技术栈需要展示其接入方式和调用边界。

## 功能范围

### 1. 首页与目录

必做能力：

- 左侧显示车辆手册目录。
- 右侧显示当前选中条目的图文详情。
- 支持一级和二级目录。
- 支持横屏布局。
- 支持当前目录项选中态。

建议目录：

- 快速开始
- 驾驶前准备
- 车灯与雨刮
- 空调与座椅
- 中控与娱乐
- 驾驶辅助
- 充电与能耗
- 保养与故障

验收标准：

- 首屏进入后无需额外跳转即可看到目录和详情。
- 宽屏下目录与详情并排显示。
- 窄屏或移动 Web 下可以退化为抽屉、上下布局或分屏切换。

### 2. 手册详情

必做能力：

- 展示标题、摘要、正文、步骤说明和图片。
- 支持长内容滚动。
- 支持图片懒加载或占位。
- 支持收藏当前条目。
- 支持记录浏览历史。

验收标准：

- 文本和图片不应在不同分辨率下重叠。
- 长内容滚动流畅。
- 收藏状态在重新进入页面后保持一致。

### 3. 全局搜索

必做能力：

- 支持按标题和正文关键词搜索。
- 支持空结果状态。
- 支持点击结果进入详情。
- 支持搜索历史或最近搜索词。

进阶能力：

- 支持关键词高亮。
- 支持拼音、英文别名或同义词。
- 支持本地索引缓存。

验收标准：

- 搜索结果与详情页数据一致。
- 断网状态下可搜索本地已缓存内容。

### 4. 收藏与浏览历史

必做能力：

- 支持添加和取消收藏。
- 支持收藏列表。
- 支持浏览历史列表。
- 支持清空历史。

验收标准：

- 应用重启后收藏和历史仍存在。
- 删除或取消收藏后 UI 及时刷新。

### 5. 图片加载与本地缓存

必做能力：

- 支持远程图片或统一资源目录图片加载。
- 支持图片占位、加载失败状态。
- 支持本地缓存。

验收标准：

- 网络正常时图片可正常展示。
- 断网或离线模式下，已缓存图片可继续展示。
- 图片加载失败不会导致页面空白或崩溃。

### 6. 离线模式

必做能力：

- 支持手动开启和关闭离线模式。
- 离线模式下优先读取本地缓存。
- 未缓存数据展示明确提示。

验收标准：

- 离线模式开关状态可见。
- 离线模式下不依赖真实网络请求完成核心浏览流程。

### 7. 深色主题

必做能力：

- 支持浅色和深色主题切换。
- 支持跟随系统主题。
- 保持文本、图标、分割线和选中态可读。

验收标准：

- 主题切换后核心页面不需要重启即可刷新。
- 不同技术栈应尽量保持视觉风格一致。

### 8. 多语言

必做能力：

- 支持中文和英文。
- 支持运行时切换语言。
- 页面标题、按钮、空状态和错误提示均应国际化。

验收标准：

- 切换语言后主要页面文本更新完整。
- 本地化文本不能撑破按钮、列表项和卡片。

### 9. 网络请求

必做能力：

- 支持从本地 Mock 服务、静态 JSON 或远程接口加载手册数据。
- 支持加载中、成功、失败三种状态。
- 支持超时或错误提示。

建议接口：

```text
GET /manual/categories
GET /manual/articles
GET /manual/articles/{id}
GET /vehicle/status
```

验收标准：

- 网络失败时不会出现不可恢复的白屏。
- 缓存策略与离线模式行为一致。

### 10. 文件选择

必做能力：

- 通过系统文件选择器选择一个文件。
- 展示文件名、大小和 MIME 类型。
- 处理用户取消选择的情况。

验收标准：

- 原生 Android、Hybrid、RN、Flutter 等技术栈都能展示各自调用方式。
- 权限或系统能力不可用时有降级提示。

### 11. 相机

必做能力：

- 调用系统相机或相册能力获取图片。
- 展示拍摄或选择后的图片预览。
- 处理权限拒绝。

验收标准：

- 权限申请流程清晰。
- 拒绝权限后应用不崩溃，并提供可理解的提示。

### 12. 通知

必做能力：

- 发送一条本地通知。
- 通知内容模拟车辆提醒，例如“胎压偏低”“充电完成”“车门未关”。
- 处理 Android 运行时通知权限。

验收标准：

- Android 13 及以上需要处理通知权限。
- 用户点击通知后可回到应用内对应页面或提醒中心。

### 13. Android Service 与模拟车辆数据

必做能力：

- 使用 Android Service 或等价模拟器持续产生车辆状态。
- 车辆状态至少包含电量、续航、胎压、车门、空调、车速。
- 页面可以订阅并展示车辆状态变化。

验收标准：

- 数据更新不会阻塞 UI。
- 页面退出或应用后台时资源释放策略明确。
- 跨平台方案需要说明其与原生 Service 的通信方式。

### 14. 横屏和多分辨率适配

必做能力：

- 支持智能座舱横屏。
- 支持平板、大屏、普通手机宽度的响应式布局。
- 避免固定像素导致内容溢出。

建议断点：

| 宽度 | 布局建议 |
| --- | --- |
| 小于 600dp | 单列布局，目录可折叠 |
| 600dp 到 960dp | 主从分区，目录较窄 |
| 大于 960dp | 稳定左右分栏，详情区展示更多信息 |

验收标准：

- 不同分辨率下文本、按钮、图片和列表不重叠。
- 横屏下主要操作不依赖过深层级。

## 技术栈验收分级

### P0：最小可比较版本

每个核心技术栈至少完成：

- 首页/目录页
- 手册详情页
- 搜索
- 车辆状态展示
- 一个原生能力调用
- 横屏适配
- 基础性能记录

### P1：完整课程版本

在 P0 基础上增加：

- 收藏和历史
- 图片缓存
- 离线模式
- 深色主题
- 多语言
- 文件、相机、通知三个原生能力

### P2：毕业项目增强版本

在 P1 基础上增加：

- 完整性能测试入口
- Hybrid 白屏治理和安全治理
- RN 或 Flutter 与 Android 原生混合导航
- KMP 共享 Repository
- 存量项目接入方案
- 多技术栈共存边界设计

## 性能测试口径

建议每个技术栈记录以下指标：

| 指标 | 记录方式 |
| --- | --- |
| 冷启动时间 | 从点击图标到首页可交互 |
| 热启动时间 | 从后台返回到首页可交互 |
| 首页首屏时间 | 首页目录和详情都可见 |
| 内存占用 | 首页稳定后、长列表滚动后分别记录 |
| 帧率 | 滚动详情页、搜索结果页、车辆数据刷新页 |
| 包体积 | Debug 与 Release 分别记录 |
| 图片加载耗时 | 首次加载与缓存命中分别记录 |
| 原生调用耗时 | 文件、相机、通知、车辆数据调用边界 |

## 非目标

为了保持课程聚焦，本项目暂不要求：

- 连接真实车辆总线或真实车机系统。
- 实现完整账号体系。
- 实现真实支付、订单或用户运营能力。
- 覆盖所有 iOS 生产发布细节。
- 追求所有技术栈功能完全等价。

课程关注的是在统一业务口径下比较不同技术方案的工程特征，而不是把每个技术栈都做成完整商业产品。
